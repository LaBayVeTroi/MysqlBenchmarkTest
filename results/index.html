MariaDB java connector performance

We always talk about performance, but the thing is always "Measure, don't guess!".

A lot of performance improvement has been done lately on the MariaDB java connector. So, what the current driver perf?

Here is benchmark result of 3 jdbc drivers permitting access to a MySQL/MariaDB database: DrizzleJDBC, MySQL connector/J and MariaDB java connector.

Driver's versions are the latest GA available version at the time of writing this blog:
  • Mariadb 1.5.0-RC1
  • Mysql5.1.39
  • Drizzle 1.2

The benchmark

JMH is an oracle micro-benchmarking framework tool developed by oracle, delivered as openJDK tools. Its distinctive advantage over other frameworks is that it is developed by the same guys in Oracle who implement JIT (Just In Time compilation). JMH is in sync with the latest Oracle JRE changes, which make its results very reliable, and permit to avoid most of micro-benchmark pitfalls.

Benchmark source: https://github.com/rusher/mariadb-java-driver-benchmark.

Tests are pretty straightforward.
Example : 

        public class BenchmarkSelect1RowPrepareText extends BenchmarkSelect1RowPrepareAbstract {

            @Benchmark
            public String mysql(MyState state) throws Throwable {
                return select1RowPrepare(state.mysqlConnectionText, state);
            }

            @Benchmark
            public String mariadb(MyState state) throws Throwable {
                return select1RowPrepare(state.mariadbConnectionText, state);
            }

            @Benchmark
            public String drizzle(MyState state) throws Throwable {
                return select1RowPrepare(state.drizzleConnectionText, state);
            }

        }

        public abstract class BenchmarkSelect1RowPrepareAbstract extends BenchmarkInit {
            private String request = "SELECT ?";

            public String select1RowPrepare(Connection connection, MyState state) throws SQLException {
                try (PreparedStatement preparedStatement = connection.prepareStatement(request)) {
                    preparedStatement.setString(1, state.insertData[state.counter++]);
                    try (ResultSet rs = preparedStatement.executeQuery()) {
                        rs.next();
                        return rs.getString(1);
                    }
                }
            }
        }

Tests using INSERT's queries are send to a BLACKHOLE engine (data are not stored). This permit to have more stable results.
(99.9% of queries execution time have 1-3% time difference, without using the blackhole engine, these execution times would vary up to 10%).

Benchmark results have been executed on MariaDB 10.1.16 and 10.2.2 and MySQL Community Server 5.7.13 databases. Results are similar (see raw results at document bottom), so the folowing document show results of MariaDB 10.2.2.

Performance comparaisons

Benchmark will tests performances of the 3 main differents behaviour using a local database (same server), and a distant database (another identical server) on same datacenter with an average ping of 0.324ms

different behaviours:

Text protocol

This correspond to option useServerPrepStmts disabled.
Query are send directly to server with parameters replacement done client side.
Data are send like text : example a timestamp will be send like text "1970-01-01 00:00:00.000500" using 26 bytes

Binary protocol

This correspond to the option useServerPrepStmts enabled (default implementation on MariaDB driver).
Data are send in binary : example timestamp "1970-01-01 00:00:00.000500" will be send using 11 bytes.

There is up to 3 exchanges with server for one query :

  1. PREPARE - Prepares statement for execution.
  2. EXECUTE - Send parameters
  3. DEALLOCATE PREPARE - Releases a prepared statement.
See Server prepare documentation for more information.

PREPARE result are stored in cache (default size 250). If Prepare is already in cache, PREPARE will not be executed, DEALLOCATE will be executed only when PREPARE is not used anymore and not in cache. That mean some query execution will have 3 round trips, but some will just have one round trip, sending a PREPARE identifier and parameters.

Rewrite

This correspond to the option rewriteBatchedStatements enabled.
Rewrite use the text protocol and concern only batches. Driver will rewrite query for faster results.

example:
Insert into ab (i) values (?) with first batch values [1] and [2] will be rewritten
Insert into ab (i) values (1), (2).

If query cannot be rewriten in "multi-values", rewrite will use multi-queries :
Insert into table(col1) values (?) on duplicate key update col2=? with values [1,2] and [2,3] will be rewritten
Insert into table(col1) values (1) on duplicate key update col2=2;Insert into table(col1) values (3) on duplicate key update col2=4

Downsides of this options are :

  • Auto increment ids cannot be retrieved using Statement.html#getGeneratedKeys().
  • Multi-queries in one execution are enable. That's not a problem for PreparedStatement, but if application use Statement that can be a security degradation (sql injection).

* MariaDB and MySQL have those 3 beheviour implemented, Drizzle only the Text protocol.

Single SELECT query

    private String request = "SELECT ?";

    public String select1RowPrepare(Connection connection, MyState state) throws SQLException {
        try (PreparedStatement preparedStatement = connection.prepareStatement(request)) {
            preparedStatement.setString(1, state.insertData[state.counter++]); //a random 100 bytes String.
            try (ResultSet rs = preparedStatement.executeQuery()) {
                rs.next();
                return rs.getString(1);
            }
        }
    }

When PREPARE results for this exact query is already in cache (cache hit), query will be faster (18.7% for the SELECT) than using text protocol. Due to the additional request PREPARE and DEALLOCATE exchanges, cache miss is 84.1% slower.
This emphasis the advantage and inconvienient of using a binary protocol. Cache HIT is important.

Using Mariadb 10.2.2 server, PREPARE + EXECUTE commands are send without waiting for PREPARE result, permitting client to avoid some network latency exchange. That make big difference using distant database, permitting to avoid most of the cache miss additional time.

Batch : 1000 INSERT query

    private String request = "INSERT INTO blackholeTable (charValue) values (?)";

    public int[] executeBatch(Connection connection, String[] data) throws SQLException {
        try (PreparedStatement preparedStatement = connection.prepareStatement(request)) {
            for (int i = 0; i < 1000; i++) {
                preparedStatement.setString(1, data[i]); //a random 100 byte data
                preparedStatement.addBatch();
            }
            return preparedStatement.executeBatch();
        }
    }

"Select 1000 rows"

private String request = "select * from seq_1_to_1000"; //using the sequence storage engine

    private ResultSet select1000Row(Connection connection) throws SQLException {
        try (Statement statement = connection.createStatement()) {
            try (ResultSet rs = statement.executeQuery(request)) {
                while (rs.next()) {
                    rs.getString(1);
                }
                return rs;
            }
        }
    }

BenchmarkSelect1000Rows.mysql                             avgt  150   551.705 ± 20.227  us/op

When using a lot of data, time is mostly spent reading from socket, so results will tend to be closer. If test was only executing the SELECT without reading results, execution time would be equivalent for mysql and mariadb (even 1% faster for mysql). Since SELECT query goal is to have results, mariadb driver is optimized to read results (avoiding the creation of buffers comparing to mysql driver).

Store procedures

Procedure call

    private String request = "{call inOutParam(?)}"; //CREATE PROCEDURE inoutParam(INOUT p1 INT) begin set p1 = p1 + 1; end

    private String callableStatementWithOutParameter(Connection connection, MyState state) throws SQLException {
        try (CallableStatement storedProc = connection.prepareCall(request)) {
            storedProc.setInt(1, state.functionVar1); //2
            storedProc.registerOutParameter(1, Types.INTEGER);
            storedProc.execute();
            return storedProc.getString(1);
        }
    }

BenchmarkCallableStatementWithOutParameter.mariadb        avgt  150    90.599 ±  5.683  us/op
BenchmarkCallableStatementWithOutParameter.mysql          avgt  150   887.335 ± 50.604  us/op

MySQL and MariaDB implementation completely differ. Mysql will use many hidden query to obtain output result. MariadDB implementation is straightForward using a capability to have OUT parameter in server response without any additional queries. (That's the main reason why MariaDB 1.4 driver require MariaDB/MySQL version 5.5.3 minimum).


Conclusion

MariaDB driver rocks !

Binary protocol has different advantages but rely on having the PREPARE results already in cache. If applications have a lot of different kind of query and database is distant, that may not be the better solution.

Rewrite has amazing results to write data in batch, and can permit to send a load a database for example
